### 1.2 Findings

#### Finding 1 — Security: `expect()` panic in `auth.rs`

| Field | Detail |
|-------|--------|
| **Severity** | Medium |
| **File** | `src/tidal/auth.rs:14` |
| **Description** | `build_base_client()` used `.expect("build HTTP client")` on `reqwest::Client::builder().build()`. If the builder fails (e.g. TLS backend unavailable), the binary panics with an unhelpful message instead of propagating the error. |
| **Fix** | Changed return type to `Result<TidalClient>`, replaced `.expect()` with `.context("build HTTP client")?`. All call sites updated to propagate with `?`. |
| **Verified** | `cargo check` passes; `tpb status` runs without panic. |

#### Finding 2 — Clippy: Collapsible `if` in `auth.rs`

| Field | Detail |
|-------|--------|
| **Severity** | Low |
| **File** | `src/tidal/auth.rs:43-48` |
| **Description** | Nested `if !force { if keychain::load()?.is_some() { ... } }` flagged by `clippy::collapsible_if`. |
| **Fix** | Collapsed to `if !force && keychain::load()?.is_some() { ... }`. |
| **Verified** | `cargo clippy -- -W clippy::all` produces 0 warnings. |

#### Finding 3 — Code Smell: `TpbError` defined but never used

| Field | Detail |
|-------|--------|
| **Severity** | Low |
| **File** | `src/error.rs` |
| **Description** | A full `TpbError` enum with 6 variants was defined, but every function in the codebase returns `anyhow::Result`. Two competing error strategies existed simultaneously. |
| **Fix** | Removed the unused enum. Replaced with a comment noting typed errors will be introduced in Phase 3+ when the codebase warrants them. |
| **Verified** | `cargo check` passes; no dead-code warning for `TpbError`. |

#### Finding 4 — Code Smell: `TpbError::Http` overlaps `TpbError::Tidal`

| **Verified** | N/A — enum removed. |

### 2.2 Keychain Tests (`src/keychain.rs`)

#### Test 1: `authz_serialise_round_trip`

| Field | Detail |
|-------|--------|
| **Rationale** | The Keychain stores `Authz` as a JSON string. If serialisation or deserialisation breaks, authentication state is lost. This is the most critical data path in the application. |
| **Input** | `Authz { access_token: "test_access_token", refresh_token: "test_refresh_token", user_id: 12345, country_code: Some("SG") }` |
| **Operation** | `serde_json::to_string()` → `serde_json::from_str()` |
| **Expected Output** | Restored `Authz` has identical field values to the original. |
| **Result** | Pass |

#### Test 2: `authz_serialise_without_country_code`

| Field | Detail |
|-------|--------|
| **Rationale** | `country_code` is `Option<String>`. Tidal may return `None` for some accounts. The serialisation must handle the `None` case without panic or data corruption. |
| **Input** | `Authz { access_token: "token", refresh_token: "refresh", user_id: 99, country_code: None }` |
| **Operation** | Serialise → deserialise round-trip |
| **Expected Output** | `user_id` is 99, `country_code` is `None`. |
| **Result** | Pass |

#### Test 3: `authz_json_contains_expected_fields`

| Field | Detail |
|-------|--------|
| **Rationale** | Guards against tidalrs upstream changes to field names via `#[serde(rename)]`. If a field name changes in the `Authz` struct, stored Keychain entries would silently fail to deserialise. |
| **Input** | Standard test `Authz` (same as Test 1) |
| **Operation** | Serialise to JSON, parse as `serde_json::Value`, check field presence |
| **Expected Output** | JSON object contains keys `access_token`, `refresh_token`, `user_id`, `country_code`. |
| **Result** | Pass |

#### Test 4: `authz_deserialise_rejects_invalid_json`

| Field | Detail |
|-------|--------|
| **Rationale** | If the Keychain entry is corrupted (e.g. partial write, manual edit), deserialisation must fail cleanly rather than producing garbage data. |
| **Input** | `"not json"` |
| **Operation** | `serde_json::from_str::<Authz>()` |
| **Expected Output** | `Err` result. |
| **Result** | Pass |

#### Test 5: `authz_deserialise_rejects_missing_fields`

| Field | Detail |
|-------|--------|
| **Rationale** | A JSON object with only some fields (e.g. just `access_token`) must not deserialise into an `Authz` with zeroed/empty required fields. Serde must enforce all required fields are present. |
| **Input** | `{"access_token":"t"}` |
| **Operation** | `serde_json::from_str::<Authz>()` |
| **Expected Output** | `Err` result (missing `refresh_token`, `user_id`). |
| **Result** | Pass |

#### Test 6: `keychain_save_load_delete_cycle` (ignored)

| Field | Detail |
|-------|--------|
| **Rationale** | End-to-end integration test verifying the full Keychain lifecycle: store credentials, retrieve them, delete them, confirm deletion. |
| **Input** | Standard test `Authz` |
| **Operation** | `save()` → `load()` → assert match → `delete()` → `load()` → assert `None` |
